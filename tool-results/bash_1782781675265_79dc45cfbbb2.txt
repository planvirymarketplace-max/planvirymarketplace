PART IV — Business Rules

Business rules are the invariants the system must enforce regardless of
which code path, user role, or API client triggers an action. Every rule
has a unique ID (BR-XXXXXX), a human description, priority, the
validation method, the failure message returned to callers, and the
affected system components.

  NOTE: This Part contains the foundational rules for Parts 0–10 scope.
  The full rule catalog (thousands of rules) will be generated in
  subsequent specification versions as each module is detailed.

Format: BR-[MODULE]-[SEQUENCE] e.g., BR-U-001 (User module, rule 001),

PART V — State Machines

Every entity with a lifecycle is governed by an explicit Finite State
Machine (FSM). States and transitions are enforced at the database layer
via triggers — application code may only call transition functions,
never write status fields directly.

5.1 Universal State Vocabulary

The following states are reserved across all entity FSMs and must not be
repurposed with different semantics in any module:

  -----------------------------------------------------------------------
  State              Meaning Across All Entities
  ------------------ ----------------------------------------------------
  DRAFT              Created but not yet visible to non-owners

  PUBLISHED / ACTIVE Visible and operational

  HIDDEN             Temporarily removed from public view by owner action

  PAUSED             Temporarily not accepting new interactions
                     (bookings, applications)

  CANCELLED          Terminated before completion — may trigger refunds
                     or reversals

  COMPLETED          Successfully finished — terminal state, no further
                     transitions

  EXPIRED            TTL elapsed without reaching a terminal state —
                     inventory released

  ARCHIVED           Soft-removed from active use; historical record
                     preserved

  DELETED            Soft-delete marker; record not destroyed, PII
                     anonymized

  SUSPENDED          Locked by platform action (moderation, policy
                     violation)

  PENDING            Awaiting an external event (payment, approval, etc.)
                     to advance

  CONFIRMED          External event received; commitment made by all
                     parties

  NO_SHOW            Guest/party failed to appear; booking completed
                     without service delivery
  -----------------------------------------------------------------------

5.5 FSM Enforcement Pattern

All state transitions are enforced via Postgres functions (RPCs) that
include the following steps, in order:

37. Acquire row lock on the target record (SELECT ... FOR UPDATE)

38. Validate current state matches expected 'from' state — reject with
    error if not

39. Validate all entry conditions for the target 'to' state

40. Execute the state write (UPDATE status = 'NEW_STATE')

41. Write audit log entry (reservation_state_log or equivalent) with
    timestamp, actor, and previous state

42. Emit domain event to the event bus (Supabase Realtime or pg_notify
    queue)

43. Return updated record to caller

  NOTE: Direct UPDATE statements targeting status columns on
  FSM-governed tables are forbidden outside of these RPC functions. A DB
  trigger rejects any status write that does not originate from the
  designated RPC.
